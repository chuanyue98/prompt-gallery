# 功能规格说明书：初始化 Prompt Gallery 基线

**功能分支**: `000-init`  
**创建日期**: 2026-04-18  
**状态**: 草案  
**输入**: 用户描述: "补充000-init/ 通过这个规格 能生成到001之前的代码。"

## 用户场景与测试 *(必填)*

### 用户故事 1 - 浏览初始画廊 (优先级: P1)

作为首次访问 Prompt Gallery 的用户，我希望进入页面后就能浏览一个具有强视觉风格的提示词画廊，支持搜索、分类筛选、卡片预览和详情查看，这样我可以快速找到感兴趣的作品并复制提示词。

**为什么选择此优先级**: 这是整个产品最核心的价值入口；如果没有稳定的浏览体验，后续投稿与同步流程都失去展示意义。

**独立测试**: 打开首页，确认可看到桌面端三列画廊、搜索与分类控件；点击卡片后可打开详情弹窗并复制提示词。

**验收场景**:

1. **假设** 用户打开首页, **当** 静态数据加载成功, **那么** 页面必须展示带有图片/视频封面的画廊卡片列表，并在标准桌面宽度下默认每行显示 3 个项目。
2. **假设** 用户输入关键词或切换图片/视频分类, **当** 列表更新时, **那么** 结果必须仅保留符合搜索词与分类条件的项目。
3. **假设** 用户点击任一作品卡片, **当** 详情弹窗打开时, **那么** 必须看到完整提示词、描述和模型信息。
4. **假设** 用户在卡片或详情弹窗中点击复制操作, **当** 浏览器支持复制能力时, **那么** 系统必须给出明确的成功反馈。

---

### 用户故事 2 - 发起社区投稿 (优先级: P2)

作为想贡献作品的创作者，我希望在网页端直接上传图片或视频，并填写作品标题、提示词与相关信息，系统自动根据标题生成语义化的文件夹路径并发起 GitHub Pull Request，这样我的作品在仓库中更具可读性。

**为什么选择此优先级**: Prompt Gallery 的内容增长依赖低门槛投稿，而清晰的目录结构是项目长期可维护的基础。

**独立测试**: 点击导航栏中的投稿入口，填写包含中文标题的表单并提交，验证服务端在 `public/data/` 下生成以“标题-5位随机数”命名的目录。

**验收场景**:

1. **假设** 用户点击“我要投稿”, **当** 投稿弹窗打开时, **那么** 页面必须允许上传图片或视频文件，并强制填写标题，同时可选填描述、Prompt、标签和模型。
2. **假设** 用户提交合法表单, **当** 服务端处理完成时, **那么** 系统必须在目标仓库中新建投稿分支，并创建路径为 `public/data/{images|videos}/{标题}-{随机码}/` 的目录，其中包含 `index.md` 与媒体文件。
3. **假设** 用户缺少标题、媒体文件或 Prompt, **当** 用户提交表单时, **那么** 系统必须阻止提交并返回明确错误信息。

---

### 用户故事 3 - 维护静态数据基线 (优先级: P3)

作为项目维护者，我希望所有画廊内容都由 `public/data/` 下的 Markdown 与媒体文件驱动，并通过统一脚本生成前端可消费的 JSON 数据，这样仓库可以保持 git-first、可审查和可迁移。

**为什么选择此优先级**: 没有稳定的数据同步脚本，前端展示与投稿 PR 都无法形成一致的数据来源。

**独立测试**: 在 `public/data/` 中新增一个合法作品目录后运行同步命令，验证 `gallery-data.json` 被正确生成，前端可读取并展示。

**验收场景**:

1. **假设** 维护者在 `public/data/images/` 或 `public/data/videos/` 中新增一条作品, **当** 运行同步脚本时, **那么** 系统必须把 `index.md` 的 Frontmatter 与正文整理为统一的静态 JSON 列表。
2. **假设** 某条作品缺少封面或主媒体文件, **当** 同步脚本执行时, **那么** 系统必须根据目录中的可用资源自动推导合理的主媒体与封面字段。
3. **假设** 前端请求静态 JSON, **当** 数据存在时, **那么** 画廊页面必须仅依赖这份 JSON 完成渲染，而不是直接扫描文件系统。

### 边缘情况

- 当 `gallery-data.json` 请求失败或没有任何可展示内容时，首页必须展示明确的空状态或错误提示，而不是白屏。
- 当作品使用视频资源但没有独立封面时，系统必须允许回退到视频文件或目录中第一张图片作为封面。
- 当投稿标题包含空格、中文或特殊字符时，系统必须生成以“标题-5位随机码”组成的文件夹路径，仅过滤掉操作系统禁止的非法路径字符，以保留中文语义和确保唯一性。
- 当作品来源于外部 URL 而非仓库内文件时，系统必须能够保留外部媒体地址并继续渲染。

## 需求 *(必填)*

### 功能需求

- **FR-001**: 系统必须提供一个具有强视觉风格的首页，包含导航、Hero 区域和画廊内容区。
- **FR-002**: 系统必须从静态 `gallery-data.json` 加载画廊数据，并支持关键词搜索与图片/视频分类筛选。
- **FR-003**: 系统必须提供画廊卡片与详情弹窗，详情弹窗中显示完整提示词、描述和模型信息。
- **FR-004**: 系统必须提供复制提示词能力，并在成功后给出可见反馈。
- **FR-005**: 初始桌面端画廊布局必须以 3 列为主要展示密度，为后续布局优化留出演进空间。
- **FR-006**: 系统必须提供网页端投稿弹窗，支持上传媒体文件并填写标题、描述、Prompt、标签和模型。
- **FR-007**: 系统必须通过服务端接口将投稿转化为 GitHub Pull Request，并写入 `public/data/{images|videos}/{slug}/index.md` 与对应媒体文件。
- **FR-008**: 系统必须使用统一同步脚本从 `public/data/` 生成前端消费的静态 JSON 数据。
- **FR-009**: 同步脚本必须同时支持仓库内文件与外部媒体 URL 两种资源来源。
- **FR-010**: 数据模型必须允许作品同时携带标题、描述、标签、Prompt 正文、模型、媒体路径和可选来源链接。

### 关键实体 *(如果功能涉及数据，请包含此部分)*

- **GalleryItem**: 代表画廊中的单条作品，包含标题、描述、标签、提示词内容、模型、媒体信息与可选来源链接。
- **ContributionDraft**: 代表一次网页投稿请求，包含待提交的元数据、媒体文件和目标 slug。
- **MediaAsset**: 代表图片或视频资源，可来自仓库内文件，也可来自外部直链。

## 成功标准 *(必填)*

### 可衡量的结果

- **SC-001**: 访客在首次打开首页后的 3 次点击以内即可完成“浏览列表 -> 打开详情 -> 复制提示词”的核心路径。
- **SC-002**: 桌面端画廊默认以 3 列呈现，且搜索与分类筛选后能即时收敛到匹配结果。
- **SC-003**: 合法投稿可在一次提交中完成媒体与元数据入库，并成功生成可审查的 Pull Request。
- **SC-004**: 维护者在新增或修改 `public/data/` 内容后，只需执行一次同步命令即可刷新前端展示数据。

## 假设

- `000-init` 不包含 `002` 的极简卡片优化，因此首页卡片仍可保留更多文字信息与快捷复制入口。
- `000-init` 不包含 `003` 的删除申请 PR 流程，因此详情页中没有“申请下架”入口。
- `000-init` 不强制要求自动化测试基线，测试补齐由后续 `004-add-test-coverage` 单独覆盖。

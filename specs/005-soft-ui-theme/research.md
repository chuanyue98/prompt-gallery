# 研究记录：Soft UI 主题切换

## 决策 1：使用语义化主题 token + `data-theme` 驱动整站主题

- Decision: 在根节点暴露统一主题标识，并将背景、文字、边框、表面、强调色、阴影和状态反馈收敛为语义化 CSS 变量，再由页面和组件消费这些变量。
- Rationale: 现有实现大量使用硬编码颜色和阴影，如果只在个别组件上叠加第二套样式，后续维护成本会迅速升高，也容易出现切换后局部残留。使用语义化 token 可以同时保留默认 cyber-obsidian 风格，并为 Soft UI 提供独立表面、阴影和动效语义。
- Alternatives considered:
  - 继续使用零散的 Tailwind arbitrary value 覆盖：改动快，但会把主题逻辑散落在各组件中，难以保证一致性。
  - 引入第三方主题库：超出当前功能所需，也违反“不引入新依赖”的约束。

## 决策 2：主题偏好使用本地持久化，并在布局层尽早恢复

- Decision: 将用户主题选择持久化在浏览器本地，并在应用布局初始化时尽早恢复，保证刷新或再次访问时沿用上次选择。
- Rationale: 主题偏好是纯前端展示状态，不需要进入 Markdown/JSON 数据流。本地持久化即可满足规格要求，同时避免把展示偏好耦合到服务端接口。
- Alternatives considered:
  - 仅保存在运行时内存：刷新即丢失，无法满足 FR-004。
  - 写入远端用户资料：当前项目没有对应用户偏好数据流，且会无必要地扩大范围。

## 决策 3：优先适配应用壳层、导航、画廊卡片、详情弹层和投稿弹层

- Decision: 第一阶段把主题适配聚焦在 `page.tsx`、`Navbar.tsx`、`Gallery.tsx`、`ContributeModal.tsx` 这些高频区域，并通过共享 token 保证未专门适配区域至少可读可用。
- Rationale: 规格明确指出主题首先服务于设定面板、小工具、控制板和个人效率工具；在当前仓库里，这些体验主要落在导航、筛选控件、卡片、详情和投稿面板中。先覆盖这些热区，最能体现主题价值，同时控制改动范围。
- Alternatives considered:
  - 一次性重写所有页面元素：范围过大，不符合增量交付原则。
  - 只改导航入口：用户能切换但无法感知主题价值，不满足用户故事 2。

## 决策 4：Soft UI 交互反馈通过“上浮/下沉”阴影语义统一实现

- Decision: 在 Soft UI 主题中，为按钮、筛选器、输入框和卡片定义默认、hover、active、focus 四类表面反馈，统一体现柔和浮雕与按压感。
- Rationale: Soft UI 的核心不是单一配色，而是表面深浅、内外阴影和状态切换的一致语言。统一交互反馈语义能让不同组件在视觉上形成同一套材质感。
- Alternatives considered:
  - 仅替换背景和边框颜色：会让新主题看起来像浅色换肤，而不是完整的 Soft UI 风格。
  - 为每个组件单独设计状态：自由度高，但一致性差，测试和维护成本也更高。

## 决策 5：验证以组件测试为主，覆盖主题切换、持久化和关键表面呈现

- Decision: 在现有 `tests/components/gallery.test.tsx` 基础上扩展主题切换与关键类名/属性断言，并结合 `lint`、`test`、`build` 做最终回归。
- Rationale: 仓库当前已有组件测试基线，主题功能主要影响前端渲染和交互，最适合从用户可见行为与关键语义类名入手验证。
- Alternatives considered:
  - 只做手工验证：无法防止后续主题回归。
  - 新增整套端到端测试：超出当前仓库现有测试体系。

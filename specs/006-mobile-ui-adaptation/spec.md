# 功能规格说明书：移动端适配优化 (Mobile UI Adaptation)

**功能分支**: `006-mobile-ui-adaptation`  
**创建日期**: 2026-04-22  
**状态**: 草案  
**输入**: 用户描述: "增加手机端适配。现在根本不好看"

## 用户场景与测试 *(必填)*

### 用户故事 1 - 移动端响应式画廊浏览 (优先级: P1)

作为一名手机用户，当我访问 Prompt Gallery 时，我希望页面能够自动适配我的屏幕尺寸，使我能够舒适地浏览和搜索灵感。

**为什么选择此优先级**: 核心体验的移动端化是项目可用的基础。目前的布局在窄屏下显示效果不佳，直接影响用户留存。

**独立测试**: 在手机浏览器或 Chrome 开发者工具模拟器（宽度 < 640px）中访问首页。

**验收场景**:

1. **假设** 用户使用宽度为 375px 的手机访问, **当** 页面加载完成时, **那么** 页面不应出现水平滚动条。
2. **假设** 用户浏览画廊, **当** 屏幕宽度减小时, **那么** 画廊网格应自动从 5 列切换为 2 列或 1 列。

---

### 用户故事 2 - 移动端详情页与操作交互 (优先级: P2)

作为一名手机用户，当我点击某个作品查看详情时，我希望首屏能快速看到提示词，图片以适当比例展示，且我可以通过点击图片来进入全屏预览模式。

**为什么选择此优先级**: 详情页是用户获取价值的关键。优化图片占用空间可以提升信息密度，全屏预览则提供了更好的视觉体验。

**独立测试**: 在移动设备和桌面端打开详情弹窗，点击图片。

**验收场景**:

1. **假设** 详情弹窗已打开, **当** 在手机端时, **那么** 图片区域高度不应超过屏幕的 40%，且下方内容立即可见。
2. **假设** 用户点击详情页中的图片, **当** 点击发生时, **那么** 系统应弹出一个全屏覆盖层展示高清大图。
3. **假设** 全屏模式已打开, **当** 用户再次点击或点击关闭按钮时, **那么** 全屏层应消失。

---

### 用户故事 3 - 移动端投稿流程适配 (优先级: P3)

作为一名移动端创作者，我希望能在手机上直接上传作品或填写投稿信息。

**为什么选择此优先级**: 完善移动端闭环，支持随时随地的分享。

**独立测试**: 在移动端触发“我要投稿”弹窗并完成模拟提交。

**验收场景**:

1. **假设** 投稿弹窗已打开, **当** 用户点击输入框时, **那么** 键盘弹出不应遮挡关键输入位置或导致布局严重错乱。

### 边缘情况

- 当屏幕旋转（横屏/竖屏切换）时，布局是否能立即自适应？
- 超长标题或标签在极窄屏幕（如 320px）下是否会导致布局撑破？

## 需求 *(必填)*

### 功能需求

- **FR-001**: 系统必须通过 CSS 媒体查询适配常见的移动端分辨率（320px 及以上），确保无水平溢出。
- **FR-002**: 导航栏（Navbar）必须提供移动端精简版布局，折叠不必要的文字或改用图标。
- **FR-003**: 画廊网格（Grid）必须是响应式的：移动端 1-2 列，平板 3 列，桌面 4-5 列。
- **FR-004**: 所有弹窗（Modal）在移动端必须支持全屏或近似全屏的展示方式，并确保关闭按钮位置固定且易点。
- **FR-005**: 交互元素（Button/Link）的点击区域必须满足移动端最小尺寸要求（建议 44x44 px）。

### 关键实体

- **响应式断点**: 定义系统在不同屏幕宽度下的布局行为集合。
- **移动端视图**: 专门针对触控和窄屏优化的 UI 呈现模式。

## 成功标准 *(必填)*

### 可衡量的结果

- **SC-001**: 在 Google Lighthouse 移动端访问性（Accessibility）评分达到 90 分以上。
- **SC-002**: 确保在 320px 宽度下页面水平偏移量为 0。
- **SC-003**: 95% 的核心交互（搜索、复制、投稿）在主流手机浏览器（Safari, Chrome）中可顺畅完成。

## 假设

- 假设用户使用支持现代 CSS 特性（如 Grid, Flexbox）的移动端浏览器。
- 假设网络环境足以加载图片和视频资源。
- 现有的“软 UI”主题风格将延续到移动端，仅做布局调整而非风格重构。

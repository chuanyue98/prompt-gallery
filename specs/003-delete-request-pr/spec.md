# 功能规格说明书：详情页删除请求流程

**功能分支**: `003-delete-request-pr`  
**创建日期**: 2026-04-18  
**状态**: 草案  
**输入**: 用户描述: "在详情里面增加一个删除功能，也是通过提交Pr来实现，不要太大太显眼"

## 用户场景与测试 *(必填)*

### 用户故事 1 - 低调的删除申请 (优先级: P1)

作为一名画廊维护者或发现问题的用户，我希望在作品详情页中有一个不显眼的入口来申请删除该作品，以便清理失效或不合规的内容。

**为什么选择此优先级**: 提供了内容管理的基础闭环，且用户明确要求 UI 要低调。

**独立测试**: 在详情模态框中寻找删除入口，验证其视觉权重是否显著低于“复制”等核心功能。

**验收场景**:

1. **假设** 我正在查看某个作品的详情弹窗, **当** 我浏览功能按钮时, **那么** 我应该能找到一个“申请删除”的链接或小按钮，其样式应为幽灵按钮（Ghost）或低饱和度文字。
2. **假设** 我点击了“申请删除”, **当** 系统响应时, **那么** 应该触发一个确认流程或直接进入自动提交流程。

---

### 用户故事 2 - 自动化删除 PR 流程 (优先级: P2)

作为一名用户，当我申请删除作品时，我希望系统能自动通过 GitHub API 创建一个 Pull Request，将对应的文件删除，从而实现透明且受控的下架过程。

**为什么选择此优先级**: 符合项目现有的“社区驱动投稿”逻辑，确保数据变更可追溯。

**独立测试**: 模拟删除操作，检查 GitHub 仓库是否生成了名为 `delete/slug-name` 的新分支及对应的 PR。

**验收场景**:

1. **假设** 我确认了删除操作, **当** 后端处理完成后, **那么** 我应该看到一条成功提示，并获得指向新创建的 GitHub PR 的链接。
2. **假设** 管理员查看 PR, **当** PR 被合并时, **那么** 该作品的文件应被物理删除，且在下次同步后从页面消失。

---

## 需求 *(必填)*

### 功能需求

- **FR-001**: 必须在详情模态框（Modal）中增加一个“删除申请”或“举报下架”的入口。
- **FR-002**: 删除入口的视觉风格必须保持“不显眼”，建议使用细体字、次要颜色或幽灵按钮样式。
- **FR-003**: 点击删除后，必须弹出一个输入框，要求用户输入“下架原因”。
- **FR-004**: 必须在后端 API 中接收并处理 `reason` 字段。
- **FR-005**: 自动化流程必须：创建新分支 -> 在新分支中删除对应的目录 -> 提交变更 -> 发起 PR。
- **FR-006**: 删除 PR 的标题应标识为 `🗑️ 删除申请: [作品标题] (原因: [用户输入内容])`。

### 关键实体 *(如果功能涉及数据，请包含此部分)*

- **DeleteRequest**: 触发删除流程的请求对象，包含作品的 `slug` 和分类。

## 成功标准 *(必填)*

### 可衡量的结果

- **SC-001**: 删除入口的点击面积或视觉权重不超过主功能按钮（复制、访问来源）的 30%。
- **SC-002**: 从用户点击确认到 GitHub PR 生成的平均耗时在 5 秒以内。
- **SC-003**: 删除 PR 包含且仅包含目标文件夹的移除操作。

## 假设

- 用户在申请删除时不需要登录（与当前投稿逻辑一致，依赖 GitHub App 机器人身份）。
- 暂时不要求用户填写删除理由，以保持流程极简（或仅作为可选）。
- 同步脚本 `sync.ts` 能正确处理文件夹缺失的情况（即物理删除后数据自动更新）。

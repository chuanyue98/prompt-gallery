# 功能规格说明书：补齐项目测试覆盖

**功能分支**: `004-add-test-coverage`  
**创建日期**: 2026-04-18  
**状态**: 草案  
**输入**: 用户描述: "为现在的项目代码补齐测试。后续的代码也都需要写测试"

## 用户场景与测试 *(必填)*

### 用户故事 1 - 为现有核心能力补齐回归保护 (优先级: P1)

作为项目维护者，我需要为当前已经存在的关键页面逻辑、公共工具函数和核心接口补齐自动化测试，以便在后续改动时尽早发现回归问题。

**为什么选择此优先级**: 现有项目没有测试基线，任何重构或功能扩展都可能在无预警的情况下破坏核心行为。

**独立测试**: 可以通过运行统一测试命令，验证公共工具逻辑、关键页面交互和接口关键输入校验是否仍按预期工作。

**验收场景**:

1. **假设** 项目存在可独立验证的公共逻辑和核心页面行为, **当** 维护者执行测试命令, **那么** 系统会自动校验这些行为并给出通过或失败结果
2. **假设** 后续有人修改了现有逻辑, **当** 修改破坏既有行为, **那么** 对应自动化测试必须失败并暴露回归点

---

### 用户故事 2 - 为后续新增代码建立测试门槛 (优先级: P2)

作为项目维护者，我需要明确后续新增或修改功能时必须同步补测试，这样测试覆盖不会只停留在一次性补齐，而能持续生效。

**为什么选择此优先级**: 如果没有明确约束，测试很容易在首次补齐后再次失效，项目会回到“只写功能不写测试”的状态。

**独立测试**: 可以通过项目脚本、文档说明或开发约束，验证贡献者在新增功能时能快速发现应补的测试入口与执行方式。

**验收场景**:

1. **假设** 开发者为项目新增功能或修复缺陷, **当** 提交代码前进行本地验证, **那么** 可以通过统一约定知道需要同步新增或更新相关测试
2. **假设** 开发者只修改功能代码但未补充对应测试, **当** 其按照项目约束自检, **那么** 能明确发现该改动尚未满足测试要求

---

### 边缘情况

- 当某些模块强依赖浏览器、Next.js 运行时或外部 GitHub API 时，系统应优先覆盖可稳定验证的行为边界，并通过 mock 隔离外部依赖
- 当现有代码缺乏可测试切分点时，系统应允许进行最小范围重构，以便在不改变用户可见行为的前提下建立测试入口
- 当接口收到缺失字段、冲突输入或无效媒体地址时，系统应通过测试验证返回明确失败结果

## 需求 *(必填)*

### 功能需求

- **FR-001**: 系统必须提供一个统一的自动化测试运行入口，使维护者可以在本地一次性执行当前项目的测试。
- **FR-002**: 系统必须集成覆盖率计算引擎（如 Vitest V8），并在每次测试运行时生成覆盖率报告。
- **FR-003**: 必须配置硬性阈值：行覆盖率、函数覆盖率、分支覆盖率均需达到 **100%**。
- **FR-004**: 当代码改动未补齐测试导致覆盖率下降或低于阈值时，测试命令必须返回非零退出码（Exit Code 1）。
- **FR-005**: 系统必须允许通过隔离外部依赖或模拟运行环境的方式，稳定验证依赖浏览器能力、框架运行时或第三方服务的代码。
- **FR-006**: 当接口收到缺失字段、冲突输入或无效媒体地址时，系统应通过测试验证返回明确失败结果。
- **FR-007**: 项目必须明确后续新增功能、缺陷修复或关键逻辑变更时需要同步新增或更新相关测试。
- **FR-008**: `package.json` 中的 `test` 脚本应默认包含覆盖率检查，以确保开发者在本地即感知到测试欠账。

### 关键实体 *(如果功能涉及数据，请包含此部分)*

- **测试基线**: 当前项目中已经被自动化测试覆盖的核心行为集合，用于定义后续变更的最小回归保护范围
- **测试约束**: 面向后续开发的约定，定义何种代码变更需要同步补测试，以及本地验证的执行入口
- **高风险逻辑区域**: 一旦出错会直接影响展示结果、用户交互或数据处理结果的模块，例如公共工具函数、画廊交互逻辑和投稿接口校验逻辑

## 成功标准 *(必填)*

### 可衡量的结果

- **SC-001**: 维护者可通过单一入口在 2 分钟内触发当前项目自动化测试。
- **SC-002**: **[硬性指标]** 项目全量代码行覆盖率（Line Coverage）必须达到 **100%**，函数覆盖率（Function Coverage）必须达到 **100%**。
- **SC-003**: 对既有核心行为的人为破坏在首次执行测试时即可被发现，而无需人工逐页回归。
- **SC-004**: **[自动化门禁]** 任何导致覆盖率低于设定阈值的代码提交，都必须在本地 `npm test` 和远程 CI 流程中被自动拦截。

## 假设

- 当前阶段优先补齐单元测试与组件层测试，不强制一次性引入端到端测试
- 对第三方网络请求、GitHub App 鉴权与浏览器 API 的验证将通过 mock 或替身实现稳定测试
- 现有 UI 样式与视觉呈现不是本次测试补齐的主要覆盖目标，重点在行为正确性与关键错误分支
- 后续“都需要写测试”的约束先通过仓库脚本、文档与开发流程固化，持续集成门禁可在后续阶段补充
